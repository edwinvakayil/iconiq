# Security Policy

## Supported Versions

Iconiq is an actively evolving open-source project, and security updates are generally provided only for the latest version of the repository. Older versions may not receive patches, dependency updates, or compatibility fixes. Using the latest version is strongly recommended.

## Reporting a Vulnerability

If you discover a security vulnerability, please report it responsibly and avoid public disclosure until the issue has been reviewed and resolved.

If GitHub Security Advisories are enabled for the repository, private disclosure is preferred. If private reporting is unavailable, vulnerabilities can be reported through the issue template below.

[Report a Vulnerability](https://github.com/edwinvakayil/iconiq/issues/new?template=issue.md)

When submitting a report, include a clear description of the issue, reproduction steps, potential impact, and any affected files, components, or dependencies. Additional technical context such as screenshots, environment details, or proof-of-concept examples can help speed up investigation.

## Response Process

Reports are typically acknowledged within 72 hours. Further updates depend on severity, complexity, and reproducibility. Valid reports are investigated and resolved as quickly as possible.

## Scope

This policy applies to the repository source code, distributed registry components, documentation-related vulnerabilities, and dependency issues affecting shipped code.

## Responsible Disclosure

Please avoid actions that could negatively impact users, contributors, infrastructure, or repository availability. This includes unauthorized data access, denial-of-service testing, large-scale automated scanning, or public disclosure before a fix is available.

## Security Philosophy

Iconiq distributes components as source code to provide implementation visibility and developer control. While this improves transparency, production-level security depends on proper integration, dependency maintenance, input validation, and secure application practices implemented by the consuming team.

## Dependency Management

Dependencies are reviewed and updated regularly. Automated tooling may be used for vulnerability scanning, dependency auditing, and supply-chain monitoring when applicable.

## Acknowledgements

Responsible disclosures that help improve the security and reliability of the project are appreciated.
